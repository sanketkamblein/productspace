The Problem

Current State Pain Points

  • Specialists spend 20+ minutes per account gathering data across multiple systems
  • No unified view of overdue items, disputes, dunning history, and payment behaviour
  • Worklists are static — no intelligent prioritisation based on risk or exposure
  • Dispute resolution is manual: AR clerks research, validate, and escalate by hand
  • Payment clearing relies on manual matching — open items age and inflate DSO
  • Context lost when switching between SAP PM, FI, credit management, and emails
The Opportunity — AR Assistant

What the Engagement Layer Enables

  • Prioritised daily worklist — AI ranks accounts by financial exposure, credit risk, and payment behaviour
  • 360° customer account view in a single Joule conversation — no system switching
  • Dispute cases created and analysed autonomously from emails or natural language
  • Smart Clearing proposes invoice-payment matches, triggered after auto-clearing runs
  • Billing Adjustment Agent triggered automatically when a dispute is accepted
  • Autonomous flows run in background; specialist acts only on recommendations
💡

What is the Engagement Layer?

A new composite UI concept for Joule that surfaces AI-generated insights, agent recommendations, and action prompts directly in the user's workflow — without requiring them to switch systems or write complex queries. The AR Assistant was one of the first solutions designed for this layer at SAP.

Key UX Requirements

What the Engagement Layer Must Deliver

Distilled from user validation sessions, customer engagement discussions, and agent capability analysis. Each requirement maps to a validated user need.

Requirement 01 · Must Have

Prioritised Worklist with Reprioritisation

"I have 200 accounts. I need to know exactly which 10 to call today — and why."
  • System-generated daily worklist ranked by AI signals
  • Signals: financial exposure, credit risk, payment behavior, aging
  • User can reprioritize based on personal preferences
  • Summary of each account pre-loaded for top accounts
Requirement 02 · Must Have

360° Account View in Joule

"I open 5 different screens to prepare for one customer call. It takes 20 minutes."
  • Top 10 overdue invoices with amounts and aging
  • Customer balance, disputes, dunning history in one view
  • Promises to Pay, resubmissions, customer contacts
  • Recommended actions surfaced inline — no navigation required
Requirement 03 · Must Have

Autonomous Dispute Analysis & Resolution Proposals

"Validating a dispute means checking the original order, the delivery, the invoice — it takes hours."
  • Dispute agent autonomously validates claims against source documents
  • Proposed resolution presented to specialist for approval
  • Billing Adjustment triggered automatically for billing-error disputes
  • Notes updated on dispute case with full reasoning
Requirement 04 · Should Have

Smart Clearing with HITL Confirmation

"There are always open items that don't clear automatically — I manually match them one by one."
  • Matching proposals for invoices, payments, and credit memos
  • Human-in-the-loop approval before clearing is posted
  • Pattern detection to propose new automatic clearing rules
  • Works in tandem with Payment Matching RPT-1 app

Design Approach

0-to-1 with a Modern Toolchain

Good UX is no longer only about process-driven decision making. The fastest way to validate a concept is to build it — and the tools now exist to do that in hours, not sprints. On this project I deliberately shifted from traditional prototyping to a fail-fast, build-real approach.

🤖

Claude — AI-Assisted Design Thinking

Used Claude throughout the design process — not just for writing, but for synthesising user research, generating interaction patterns, stress-testing flows, and building working HTML/CSS prototypes that could be validated with real stakeholders immediately.

Concept Generation Rapid Prototyping Flow Validation
🎨

Figma — Design System & Handoff

Figma remains the source of truth for UI design — but used differently: as a validation layer after concepts are proven in code, not as the starting point. Allows faster iteration with engineering and reduces rework at handoff.

UI Design Design System Engineering Handoff
⚡

Make (no-code) — Workflow Automation

Used Make to simulate agent orchestration flows and automate test scenarios — enabling live demonstration of multi-agent behavior without waiting for engineering implementation. Validates the interaction model before a single line of production code is written.

Agent Flow Simulation No-Code Automation Validate Before Build
💻

Cursor / VS Code + Cline — Code as a Design Tool

Using AI-powered code editors to build working prototypes from design intent — closing the gap between concept and testable reality. This portfolio page itself was built this way. Treats code fluency as a UX skill, not an engineering handoff.

AI-Assisted Development Working Prototypes UX as Code
🤝

Designing for Agents — Not Just Users

The biggest UX shift on this project: designing for AI agents as well as people. Agent UX requires explicit design for both success paths and failure states — what an agent communicates when it cannot complete a task is as critical as what it shows when it succeeds. Getting both right reduces complexity for the user and cuts time-to-resolution significantly.

Success + Failure States Agent Communication Patterns Reduced User Complexity

The philosophy: Experience design has two modes. The first is process-driven — research, synthesis, validate, iterate. The second is fail-fast — build something real, put it in front of people, learn from what breaks. Neither is better. The designer who can switch between both, and who has the toolset to execute in both modes, moves faster and makes better products. The modern toolchain — Claude, Figma, Make, Cursor — collapses the distance between idea and working solution, making the second mode available to designers for the first time.

Learnings & Reflection

What I Learned Over 8 Months

Working at the frontier of enterprise AI — designing for autonomous agents, Joule orchestration, and a brand-new Engagement Layer concept — surfaced lessons that shaped both the product and my approach.

Key Learnings from the AR Assistant Project

Designing multi-agent systems for enterprise finance required rethinking how UX works when intelligence is distributed across many autonomous agents.

Action-first, not insight-first

Early designs surfaced a lot of data. User validation showed specialists wanted recommended actions — not more information to process. The Engagement Layer had to lead with "what to do next."

Autonomous agents need explainability

Specialists were skeptical of AI-driven prioritisation until they understood the signals behind it. Adding visible reasoning ("prioritised due to broken promises and overdue amount") was critical for trust.

Cross-assistant integration is the hardest UX problem

The AR ↔ Billing Assistant handoff for dispute resolution required careful design of notification, context transfer, and HITL confirmation. Invisible cross-system orchestration breaks user mental models.

Private Cloud constraints shape design early

Volume limitations in Private Cloud (too many open items for LLM context windows) forced architecture decisions that affected UX — especially for Smart Clearing. Platform constraints must be surfaced to UX in week one.

The Engagement Layer is still being defined

Designing for a platform still under active development means the UX patterns had to be co-created with platform teams. Being embedded in that conversation early gave the AR Assistant disproportionate influence on the pattern library.

Customer engagement drives better scope decisions

Direct involvement in customer sessions prevented over-engineering. Real users care about DSO and time-to-resolution — not about which agent topology achieves it. Keeping user outcomes central prevented feature creep.

Designing for agents is a new UX discipline

You don't just design for the user — you design for how the agent communicates with the user. Success states are easy. Failure states are where most agent UX breaks down. Explicitly designing both — what the agent says when it can't complete a task, how it recovers, what it asks the user — reduces complexity and builds trust faster than any onboarding flow.
