00 — The Concept

Why Industrial Systems?
From objects to systems thinking

Industry runs on shared language. When a maintenance engineer, a contractor, and a reliability manager look at the same asset, they must see the same thing — or errors compound into failures. Industrial Systems is built on that belief.

🌐

One Language Across the Organisation

ISO 81346-based Reference Designation System (RDS) gives every asset a unique, structured identity — readable by engineers, contractors and planners the same way, globally. No more spreadsheet-based naming conventions per site.

🔗

Systems, Not Just Objects

Traditional EAM managed Plants, Functional Locations and Equipment as isolated hierarchies. Industrial Systems treats them as interconnected nodes in a multi-perspective model — because real industrial assets are never isolated.

📐

International Standards at the Core

IEC 81346-1:2009 and the READI JIP Information Modelling Framework (IMF) define the backbone. This isn't custom logic — it's the same structural language used in nuclear, oil & gas, and power generation industries worldwide.

⚙️

Clarity for Maintenance Master Data

The persona driving this — RITA, a 30+ year Maintenance Master Data Specialist — needs high-quality, consistent data that all stakeholders trust. Industrial Systems is built around her pain points: duplicates, unclear ownership, no change impact visibility.

The Aspect Model — IEC 81346

Multiple views. One system.

Every industrial object can be described from multiple perspectives simultaneously. These are called Aspects. The same physical pump is a function (what it does), a product (what it is made of), and a location (where it lives). Function, Product and Location are the most common — but additional aspects can exist.

Crucially, each aspect can have more than one independent hierarchy. A plant may have two separate Functional aspect trees — one for process systems, one for utility systems — each with its own top node, each a complete and valid breakdown in its own right. Aspects are not limited to three, and top nodes within an aspect are not limited to one.

Take a Diesel Generator on a North Sea oil platform. The same physical unit appears across multiple aspects — none is the "primary" record. They are facets of one object. If the plant has two separate power generation hierarchies, this generator appears as a node in each Functional tree independently.

Old EAM Objects → New System Object Nodes

Every legacy Technical Object has a place in the new model. No object is lost — they become typed nodes within one or more aspects. A node may appear empty in some aspects — that is intentional clustering, not missing data.

Key insights:
1. An empty node in one aspect is not a gap — it is valid, meaningful clustering. A Plant Section with no Product aspect simply means "this location group has no product-level definition yet."
2. A single aspect type is not limited to one hierarchy. Two separate Functional trees — e.g. "Process Systems" and "Utility Systems" — can each have their own top node within the same Function aspect. An Equipment node can appear in both independently.

- (Product)
Product Aspect
Describes what the system is made of — the physical, procurable components. Assembly relationships: mounted-in, mounted-on, contains.
  • Generator Assembly → Engine → Fuel Pump
  • Reactor Vessel → Pressure Relief Valve
  • Pipeline → Segment → Flange
Technical Object mapping:
Equipment → now a node in the Product tree. Equipment "EQ-7742: Cat C18 Generator" becomes product node -GEN01 with full classification and characteristics.
+ (Location)
Location Aspect
Describes where an asset is physically located. Spatial and organisational containment — from plant to platform to room to mounting position.
  • North Sea Field → Platform → Deck
  • Plant Section → Room → Bay → Slot
  • Sub-location → Position → Mounting Point
Technical Object mapping:
Plant & Plant Section → now top nodes in the Location tree. "Troll C (Plant 1010)" becomes location node +TRL-C. All sub-locations nest below it.

How aspects connect

Aspects are never truly isolated. Relationships inside an aspect (intra) build the hierarchy. Relationships between aspects (inter) bind function to product to location — creating the complete system model.

01 — The Problem

One product. Two clouds.
Many moving parts.

SAP's Industrial Systems module needed a unified, scalable UI to manage technical systems — Equipment, Functional Locations, and their interdependencies — across fundamentally different deployment environments.

🔀

Cloud Stream Fragmentation

Private and Public Cloud teams operated on separate roadmaps, creating inconsistent UI patterns and duplicated design effort across deployment models.

🕸️

Complex System Connectivity

Industrial systems are highly interconnected. Visualizing these relationships — equipment hierarchies, system links, APM dependencies — required new interaction patterns beyond standard list views.

📊

Master Data Complexity

Asset data quality and governance directly impacted user trust. Incomplete or inconsistent master data created friction in workflows and eroded confidence in the system.

🧩

Aligning Product & Design

With no established UX voice in early backlog planning, design decisions were often made reactively. The team needed a proactive design presence to shape requirements upstream.

04 — System Topology

One system.
Many aspect instances.

A single industrial system doesn't live in one place. It fans out into multiple aspect instances — each sub-system described independently across Function, Product, and Location lenses, per domain. The same Excavators Object exists simultaneously as a Breaking Function in Oil & Gas, a Breaking Function in Construction, a Hydraulics Function, an Electric Component, and more.
